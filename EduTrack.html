Technical Implementation

PostgreSQL - Database Schema

SQL 
-- Main attendance tracking table
CREATE TABLE attendance_records (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    student_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    class_id UUID NOT NULL REFERENCES classes(id) ON DELETE CASCADE,
    qr_session_id UUID NOT NULL REFERENCES qr_sessions(id),
    attendance_time TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    status VARCHAR(20) DEFAULT 'present' CHECK (status IN ('present', 'absent', 'late', 'excused')),
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- QR session management table
CREATE TABLE qr_sessions (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    class_id UUID NOT NULL REFERENCES classes(id) ON DELETE CASCADE,
    teacher_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    qr_code TEXT NOT NULL,
    is_active BOOLEAN DEFAULT true,
    expires_at TIMESTAMP WITH TIME ZONE NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Enable Row Level Security
ALTER TABLE attendance_records ENABLE ROW LEVEL SECURITY;
ALTER TABLE qr_sessions ENABLE ROW LEVEL SECURITY;

-- RLS Policies for data security
CREATE POLICY "Teachers can view their class attendance" 
ON attendance_records FOR SELECT 
USING (EXISTS (
    SELECT 1 FROM classes 
    WHERE classes.id = attendance_records.class_id 
    AND classes.teacher_id = auth.uid()
));

-- Index for performance optimization
CREATE INDEX idx_attendance_student_class ON attendance_records(student_id, class_id);
CREATE INDEX idx_attendance_date ON attendance_records(attendance_time);
CREATE INDEX idx_qr_sessions_active ON qr_sessions(is_active, expires_at);

-- Function to mark attendance
CREATE OR REPLACE FUNCTION mark_attendance(
    p_student_id UUID,
    p_qr_code TEXT
) RETURNS JSON AS $$
DECLARE
    v_session RECORD;
    v_result JSON;
BEGIN
    -- Find active QR session
    SELECT * INTO v_session 
    FROM qr_sessions 
    WHERE qr_code = p_qr_code 
    AND is_active = true 
    AND expires_at > NOW();
    
    IF v_session IS NULL THEN
        RETURN json_build_object('success', false, 'message', 'Invalid or expired QR code');
    END IF;
    
    -- Check if attendance already marked
    IF EXISTS (
        SELECT 1 FROM attendance_records 
        WHERE student_id = p_student_id 
        AND qr_session_id = v_session.id
    ) THEN
        RETURN json_build_object('success', false, 'message', 'Attendance already marked');
    END IF;
    
    -- Insert attendance record
    INSERT INTO attendance_records (student_id, class_id, qr_session_id)
    VALUES (p_student_id, v_session.class_id, v_session.id);
    
    RETURN json_build_object('success', true, 'message', 'Attendance marked successfully');
END;
$$ LANGUAGE plpgsql;

System Workflow

1

QR Code Generation

Teacher generates a unique QR code for each class session with a specific expiration time, ensuring security and preventing reuse.

2

Attendance Scanning

Students scan the QR code using their mobile devices, which sends their student ID and QR code data to the backend for validation.

3

Data Validation

Backend validates the QR code, checks expiration, verifies student enrollment, and prevents duplicate attendance entries.

4

Record Creation

Validated attendance data is stored in the database with timestamps, creating a secure and tamper-proof attendance record.

5

Real-time Updates

Dashboard updates in real-time using Supabase subscriptions, showing current attendance status and statistics instantly.

Challenges & Solutions

Concurrent Attendance Scans

When multiple students scan the QR code simultaneously, we needed to prevent duplicate attendance records and ensure data consistency without performance bottlenecks.

Solution:

Implemented database transactions with optimistic locking and unique constraints. Used PostgreSQL's advisory locks for high-concurrency scenarios and implemented rate limiting per student.

Security Concerns

QR codes could be screenshotted and shared, allowing unauthorized attendance marking. We needed to prevent abuse while maintaining usability.

Solution:

Implemented short-lived QR codes with 5-minute expiration, IP address tracking, and device fingerprinting. Added rate limiting and suspicious activity detection algorithms.

Real-time Data Performance

The dashboard needed to display real-time attendance data without performance degradation, even with hundreds of concurrent users accessing the system.

Solution:

Used Supabase's real-time subscriptions for live updates, implemented database materialized views for complex aggregations, and added caching for frequently accessed data.
